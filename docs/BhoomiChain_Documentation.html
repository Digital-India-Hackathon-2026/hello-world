01 What is BhoomiChain?

🔴 The Problem

In India, land ownership records are still maintained on paper in many states. This leads to:

  • Forged documents — criminals photoshop land title deeds to steal property
  • Double-selling — the same plot is sold to two different buyers simultaneously
  • Ownership disputes — decades of unclear, contradictory records
  • Corruption — government officials can tamper with registry books

🟢 Our Solution

BhoomiChain replaces the paper registry with a blockchain-backed digital system where:

  • Every land parcel is represented as an NFT (Non-Fungible Token) on the Ethereum blockchain
  • Ownership is cryptographically proven — no one can forge it
  • Transfers require three-party approval (seller + buyer + government registrar)
  • An AI engine scans uploaded documents for signs of forgery before approval
  • Anyone can publicly verify who owns any piece of land, instantly
💡 The Core Idea in One Sentence

Each piece of land becomes an NFT. Whoever holds the NFT on the blockchain is the legal owner. Transferring land means transferring the NFT — but only after the seller, buyer, and government all approve.

03 Architecture Overview

BhoomiChain has a dual-layer architecture. Think of it like having two record books that always stay in sync:

⛓️ Layer 1: The Blockchain (Source of Truth)

This is the unbreakable record. The smart contract on the blockchain stores WHO owns WHAT. Nobody can change it without proper authorization. It's slow but 100% tamper-proof.

💾 Layer 2: The Database (Fast Mirror)

The SQLite database is a fast copy of the blockchain data. It stores extra details (addresses, coordinates, documents) that would be too expensive to store on-chain. The UI reads from here for speed.

How They Stay in Sync

When something happens on the blockchain (e.g., a parcel is registered), the Simulator (or Indexer in production) automatically writes the same data into the SQLite database. This way, the website loads instantly from the database, but the blockchain remains the ultimate authority.

The Simulator — Our Secret Weapon for Demos

Instead of requiring a live Hardhat blockchain node running (which can crash during a demo), Dev B built a Simulator that mimics the blockchain entirely inside SQLite tables (SimParcel, SimTransfer, ChainEvent). When judges visit the app, everything "just works" without needing to start any blockchain software.

04 The Smart Contract

The file contracts/contracts/LandRegistry.sol is the blockchain brain. Here's what it does:

What is an NFT in Our Context?

Each land parcel is minted as an ERC-721 NFT (Non-Fungible Token). The token ID = the parcel ID. Whoever "holds" the token on the blockchain is the legal owner. You cannot copy, duplicate, or forge an NFT — the blockchain guarantees uniqueness.

Roles (Who Can Do What)

RoleWhoPowers
DEFAULT_ADMINGovernment Admin (deployer)Grant/revoke registrar roles
REGISTRAR_ROLEGovernment RegistrarRegister new parcels, finalize transfers
Owner (NFT holder)Citizen who owns landInitiate transfers of their own parcels

Data Structures Stored On-Chain

Parcel (Land Record)

id (unique number) · surveyNumber (e.g., "142/2A") · district (e.g., "Pune") · geo (latitude,longitude) · area (square feet) · documentHash (SHA-256 fingerprint of the title deed PDF) · registeredAt (timestamp) · status (Active or InTransfer)

TransferRequest (Ownership Change Request)

parcelId · seller · buyer · newDocumentHash · sellerApproved · buyerApproved · registrarApproved · status (PendingBuyer → PendingRegistrar → Completed/Rejected) · createdAt

Smart Contract Functions

FunctionWho Can CallWhat It Does
registerParcel()Registrar onlyCreates a new land NFT and assigns it to an owner
initiateTransfer()Current owner onlyStarts a sale request. Locks the parcel (status → InTransfer)
buyerApprove()Designated buyer onlyBuyer agrees to purchase. Status moves to PendingRegistrar
registrarFinalize()Registrar onlyGovernment approves. NFT is transferred to buyer. Done!
rejectTransfer()Buyer, seller, or registrarCancels the transfer. Parcel goes back to Active

Events (Blockchain Notifications)

Every time something happens on the contract, it emits an Event — a permanent log entry on the blockchain that anyone can read:

  • ParcelRegistered — a new land parcel was created
  • TransferInitiated — seller started a sale
  • TransferApproved — someone (seller/buyer/registrar) approved
  • TransferCompleted — ownership officially changed
  • TransferRejected — transfer was cancelled with a reason
🔒 Anti-Theft Protection

The standard ERC-721 transferFrom() and safeTransferFrom() functions are permanently disabled (they revert with "Use transfer workflow"). This means nobody can bypass the 3-party approval system and just steal someone's land NFT.

06 The Simulator

The file web/src/lib/simulator.ts is our hackathon secret weapon.

Why Do We Need It?

A real Ethereum blockchain requires running npx hardhat node in a terminal. If that terminal crashes during a demo to judges, our entire app breaks. The Simulator eliminates this risk by faking the blockchain entirely inside SQLite.

How It Works

1

Every chain function (e.g., getParcel()) first calls shouldUseSimulator()

2

If no Hardhat node is running OR no contract address is set → Simulator mode activates

3

Instead of talking to the blockchain, the function reads/writes from SimParcel and SimTransfer tables in SQLite

4

Every simulated action also creates a ChainEvent row — so the timeline history works identically

✨ The Brilliance

From the user's perspective, the app looks and behaves identically whether it's running against a real blockchain or the simulator. The UI doesn't know the difference.

10 The Complete Transfer Workflow

This is the core feature of BhoomiChain. Here's exactly what happens when land changes hands:

1

Seller Initiates Transfer

The seller goes to /portal/transfer, enters the parcel ID and buyer's wallet address, and clicks "Initiate Transfer".

What happens behind the scenes:

  • The frontend calls POST /api/chain with action: "initiate"
  • The simulator verifies: Is this person the owner? Is the parcel Active (not already in a transfer)?
  • If valid, it creates a SimTransfer record with status PendingBuyer
  • The parcel status changes from "Active" to "InTransfer" (blocks double-selling)
  • A ChainEvent of type "INITIATED" is created
  • A TransferMeta row is created in the database
  • The fraud engine runs scanTransfer() to check for anomalies
2

Buyer Reviews and Approves

The buyer goes to /portal/approvals and sees the pending request.

  • Clicking "Review & Sign" calls POST /api/chain with action: "buyerApprove"
  • The simulator verifies: Is this the designated buyer? Is status PendingBuyer?
  • Status changes to PendingRegistrar
  • A ChainEvent of type "APPROVED" is created

If the buyer rejects: Status changes to Rejected and the parcel goes back to Active.

3

Registrar Verifies and Finalizes

The government registrar goes to /dashboard/transfers.

  • They can click "Run AI Integrity Check" to analyze the uploaded document for forgery
  • If the document is flagged as LIKELY_FORGED (risk score ≥ 70), the system blocks finalization
  • If clean, clicking "Finalize on Chain" calls POST /api/chain with action: "registrarFinalize"
  • The simulator: marks the transfer as Completed
  • The simulator: changes the parcel's ownerWallet to the buyer
  • The simulator: updates the parcel's documentHash to the new document
  • The simulator: sets parcel status back to "Active"
  • A ChainEvent of type "TRANSFERRED" is created
  • ParcelMeta.ownerWallet is updated in the database
✅ After Completion

The buyer is now the legal owner. If anyone visits /verify/[id], they'll see the buyer as the current owner, with the complete ownership history showing the transfer event.

11 Security & Fraud Detection

Smart Contract Security

  • Role-Based Access Control — Only registrars can register parcels and finalize transfers. Only owners can initiate sales. Only designated buyers can approve.
  • Anti-Bypass — Standard NFT transfer functions (transferFrom) are permanently disabled. The ONLY way to move ownership is through the multi-party workflow.
  • Double-Sale Prevention — When a transfer starts, the parcel status changes to "InTransfer". No second transfer can be initiated on the same parcel.
  • State Machine Enforcement — The transfer must go through PendingBuyer → PendingRegistrar → Completed in order. You can't skip steps.

Fraud Detection Engine

The file web/src/lib/fraud.ts implements automated fraud scanning:

Fraud TypeWhat It DetectsSeverity
DOUBLE_SALEAttempting to sell a parcel that already has an active transferHIGH
OWNERSHIP_MISMATCHThe person initiating the transfer is not the actual on-chain ownerHIGH
FORGED_DOCThe AI analysis flagged the uploaded document as likely forgedHIGH
RAPID_FLIPThe same parcel changing hands suspiciously fast (future implementation)MED
🛡️ How Scanning is Triggered

Every time a transfer is initiated or mirrored to the database, scanTransfer() runs automatically. It checks ownership, looks for active transfers on the same parcel, and analyzes any uploaded documents. Alerts appear in the Registrar's Fraud War-Room dashboard in real-time.

12 AI Document Verification

The endpoint POST /api/ai/verify-doc analyzes uploaded land title documents for signs of tampering.

How It Works

1

A document is uploaded via POST /api/upload. Its SHA-256 hash is computed and stored.

2

The registrar clicks "Run AI Integrity Check" on the transfer queue page.

3

The system compares the document's hash against the on-chain documentHash. If they don't match → instant LIKELY_FORGED verdict.

4

If hashes match, a forensic analysis checks for: font consistency, seal integrity, digital artifacts, and numeral tampering.

The AiReport Object

{ "hashMatch": true, // Does the file hash match on-chain anchor? "sha256": "0xabc...", // SHA-256 fingerprint of the document "riskScore": 18, // 0 = safe, 100 = definitely forged "verdict": "LIKELY_GENUINE", // or "SUSPICIOUS" or "LIKELY_FORGED" "indicators": { "fontConsistency": "Consistent", "sealIntegrity": "Intact", "digitalArtifacts": "None detected", "numeralTampering": "None detected" }, "reasons": ["No significant forgery indicators detected"], "model": "bhumitrust-forensic-v1" }

15 Glossary of Terms

TermSimple Explanation
BlockchainA digital ledger that records transactions. Once written, data cannot be changed or deleted. Think of it as an indestructible diary.
Smart ContractA program that runs on the blockchain. It executes automatically when conditions are met. Our LandRegistry.sol is a smart contract.
NFT (ERC-721)A unique digital token. Each land parcel is represented as one NFT. The token cannot be duplicated — there's only one of each.
Wallet AddressA unique identifier (like a bank account number) for a blockchain user. Looks like 0x70997970C518...
MetaMaskA browser extension that acts as your blockchain wallet. It stores your private keys and lets you sign transactions.
SignerAn object in ethers.js that represents a wallet capable of signing (authorizing) transactions.
SHA-256A cryptographic hash function. It takes any input (a file, text) and produces a unique 64-character fingerprint. If even one byte changes, the fingerprint is completely different.
ParcelA piece of land. In our system, each parcel has a unique ID, survey number, location, area, and owner.
ULPINUnique Land Parcel Identification Number — a government-assigned 14-digit ID for each plot of land in India.
Survey NumberA local identifier assigned by the revenue department to each piece of land within a village/district.
PrismaA tool that lets us interact with the database using TypeScript code instead of writing raw SQL queries.
SQLiteA lightweight database stored as a single file (dev.db). No server needed.
SimulatorOur custom code that pretends to be a blockchain. It stores data in SQLite tables instead of a real Ethereum node.
ABIApplication Binary Interface — a JSON file that tells our JavaScript code how to talk to the smart contract.
HardhatA development tool for Ethereum. It compiles Solidity code, runs tests, and can spin up a local blockchain for testing.
GasThe "fee" paid to execute a transaction on the blockchain. On our local Hardhat node, gas is free.
